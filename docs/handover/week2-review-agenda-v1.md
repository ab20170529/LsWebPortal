# Week 2 Review Agenda V1

This document is a practical agenda for the main week-2 review meeting.

Use it when reviewing the three target surfaces:

- system gate
- project shell
- project ledger

---

## 1. Meeting Goal

The goal of the week-2 review is not to approve polished visuals.

The goal is to confirm:

1. structure
2. hierarchy
3. role fit
4. key product decisions

If those four are stable, visual design can proceed with low rework.

---

## 2. Suggested Attendees

- PM / UI owner
- front-end engineer
- back-end engineer if permission or data assumptions matter
- business stakeholder or product approver

---

## 3. Recommended Time Box

Total suggested duration:

- 45 to 60 minutes

Suggested split:

1. 10 min: objective and context
2. 10 min: system gate
3. 10 min: project shell
4. 10 min: project ledger
5. 10 to 20 min: open decisions and alignment

---

## 4. Agenda Structure

### Part 1: Context

Explain briefly:

- week-1 findings
- week-2 objective
- why these three pages were prioritized

### Part 2: System Gate Review

Review focus:

1. business context first
2. system choice second
3. project card clarity
4. locked vs ready behavior

Questions:

1. Is company selection dependency clear enough?
2. Does this reduce confusion for first-time users?
3. Should Project receive stronger emphasis?

### Part 3: Project Shell Review

Review focus:

1. role-aware grouping
2. current project visibility
3. navigation hierarchy
4. admin/config de-emphasis

Questions:

1. Does this shell work for managers and members equally well?
2. Are the proposed first-level groups acceptable?
3. Should Admin and Standards remain top-level?

### Part 4: Project Ledger Review

Review focus:

1. project entry workflow
2. list/detail relationship
3. initialization prominence
4. permission clarity

Questions:

1. Is initialization-by-type visible enough?
2. Does the selected detail panel help continuation of work?
3. Should viewers see the same detail structure?

### Part 5: Open Decisions

Use:

- `week2-open-decisions-v1.md`

Try to close:

1. role default landing
2. Project emphasis on system gate
3. Admin/Standards top-level visibility
4. viewer default landing

---

## 5. Required Outputs From Review

By the end of the meeting, record:

1. accepted structure
2. rejected options
3. open items
4. owner for each open item
5. whether high-fidelity design can start

---

## 6. Review Rule

Do not let the meeting drift into detailed color/style debate if structural decisions are still open.

Sequence:

1. structure
2. role logic
3. state completeness
4. copy clarity
5. visual polish later
