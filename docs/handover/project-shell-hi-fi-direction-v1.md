# Project Shell Hi-Fi Direction V1

This document defines the visual direction for the Project product shell.

It translates the approved shell structure into a high-fidelity design mindset.

---

## 1. Shell Personality

The shell should feel:

- composed
- focused
- role-aware
- slightly premium, but work-first

It should not feel:

- over-styled
- noisy
- dashboard-heavy by default

The shell is not the hero.

The work inside the shell is the hero.

---

## 2. Visual Role Of The Shell

The shell should do four jobs:

1. orient the user
2. expose the current project context
3. organize work by intent
4. get out of the way

That means:

- navigation must be readable
- active workspace must be clear
- current project must be persistent
- content area must remain dominant

---

## 3. Header Direction

The header should feel:

- precise
- informative
- lightweight

### Important header elements

1. current product/workspace path
2. current project chip
3. search
4. utility actions

### Emphasis rule

- current project chip should visually outrank utility buttons
- utility buttons should stay quiet

---

## 4. Left Navigation Direction

The nav should express role logic, not just feature inventory.

### Group treatment

- first-level groups should be visually distinct
- active group should feel intentional
- inactive groups should remain easy to scan

### Item treatment

- active item should be unmistakable
- hover state should be gentle and precise
- Admin and Standards should feel secondary to daily work

### Mood

- dependable
- structured
- not playful

---

## 5. Current Project Context Direction

This is one of the most important shell elements.

### It should communicate

1. which project is currently active
2. whether the user is anchored correctly

### Design rule

- current project should be visible in both header and page context when needed
- if no project is selected, the empty state should be noticeable but not disruptive

---

## 6. Content Area Direction

The workspace content area should feel like:

- the main work surface

The shell should frame it, not compete with it.

### Design implication

- avoid over-decorating the shell chrome
- use calmer surfaces in shell areas
- allow content pages to hold stronger emphasis

---

## 7. Role-Aware Mood Differences

The shell frame can remain consistent, but its starting emphasis changes by role.

### Manager

Mood:

- control
- delivery
- oversight

Default emphasis:

- Project Control group

### Member

Mood:

- action
- clarity
- low noise

Default emphasis:

- My Work group

### Viewer

Mood:

- understanding
- read-only confidence

Default emphasis:

- Insight or Project Control

---

## 8. Tabs And Secondary Navigation

Tabs can remain, but they should not overpower the main IA.

### Rule

- left nav is primary orientation
- tabs are secondary continuity

### Visual implication

- tabs should not be louder than the active nav item
- tab accumulation should not visually dominate the header zone

---

## 9. Search Treatment

Search should feel useful but quiet.

It is not the primary action of the shell.

Use:

- calm input treatment
- clear placeholder
- no visual over-emphasis

---

## 10. Exit Action Treatment

Exit System should remain available, but not read as destructive work loss.

It is a platform navigation action.

Design rule:

- visible
- clearly separated from business actions
- low-to-medium visual weight

---

## 11. Color And Surface Strategy

### Shell Areas

Use:

- calmer surfaces
- lower visual contrast
- softer structural separation

### Active Areas

Use:

- stronger emphasis for selected nav item
- clear current project cue

### Content Area

Should maintain the strongest usable focus.

---

## 12. High-Fidelity Review Questions

1. Does the shell help orientation without dominating the page?
2. Is current project context visible enough?
3. Can members and managers both identify their correct zone quickly?
4. Are Admin and Standards properly de-emphasized?

---

## 13. Finish Line For This Page

The shell is ready when:

1. orientation is immediate
2. role entry feels logical
3. selected project context is persistent
4. content remains primary
