# 项目管理平台现状盘点与实施计划 v1

## 1. 目的

本文档用于统一当前 `LsERPPortal/project` 模块的现状、目标边界和后续实施顺序，作为下一阶段项目管理平台开发的执行基线。

本版结论以当前已实现代码和最新业务口径为准。

## 2. 已确认的业务边界

### 2.1 总体原则

- 项目管理平台继续在 `LsERPPortal` 中建设。
- 甘特图管理界面是“管理员制定计划、调整排期、分配节点任务”的主工作台。
- 涉及成员填报、任务接收、完成说明、延期申请、月周日计划、总结、日志的能力，不放在甘特管理页内，单独做菜单。
- 里程碑模板管理不另建新体系，直接复用并完善当前“项目类型 + 节点模板 + 任务模板 + 模板排期”。
- 分析中心单独建设，展示型甘特图放在上半部分，分析图表放在下半部分。

### 2.2 八大功能归属

| 功能 | 归属页面 | 页面形态 |
| --- | --- | --- |
| 1. 里程碑模板管理 | 里程碑模板管理 | 表格 + 可视化模板排期 |
| 2. 建立项目任务（项目、团队成员、费用预算） | 项目台账 | 表格管理 |
| 3. 建立项目里程碑节点（节点、任务分配） | 排期协同 | 可视化甘特 |
| 4. 成员项目任务接收、完成说明、延期申请 | 任务填报 / 延期申请 | 表格 + 表单 |
| 5. 月到周、周到日计划、总结、日志 | 计划与日志 | 表格 + 表单 |
| 6. 里程碑节点延期申请及调整 | 排期协同 + 延期申请 | 甘特调整 + 独立申请 |
| 7. 甘特图 BI 展示管理 | 项目分析看板 | 展示型甘特 + 图表 |
| 8. 项目数据分析 | 项目分析看板 | 图表 + 汇总表 |

## 3. 当前项目代码现状

### 3.1 前端现状

当前 `packages/products/project/src` 已有以下核心页面和组件：

- `project-management-page.tsx`

  - 已实现项目新增、编辑、删除
  - 已实现按项目类型初始化
- `project-type-management-page.tsx`
  - 已实现项目类型、节点模板、任务模板管理
  - 已实现模板排期配置
  - 已具备“里程碑模板管理”的基础骨架
- `project-gantt-workspace-page.tsx`
  - 已实现项目排期协同主界面
  - 已实现左侧树 + 中间甘特区
  - 已实现节点/任务新增
  - 已实现拖动改期、缩放改期、计划时间编辑
  - 已实现任务依赖与连锁更新
- `gantt-detail-panel.tsx`
  - 已实现甘特侧边详情面板
  - 已承载任务和节点局部编辑
- `project-workspace-shell.tsx`
  - 已实现上头部、左菜单、中内容区、tab 工作区框架

当前工作区定义：

- `project-management`
  - 当前名称接近“项目台账”
- `progress-config`
  - 当前实际承载“项目类型/节点模板/任务模板/模板排期”
- `project-gantt-workspace`
  - 当前实际承载“排期协同”

### 3.2 后端现状

- `projecttype`

- 任务依赖管理
- 计划、汇报、打卡、附件基础接口

## 4. 当前能力与目标能力对比

### 4.1 已具备的能力

- 系统级项目管理平台入口已搭好
- 项目台账已可维护项目主档
- 项目类型管理已具备里程碑模板管理基础
- 排期协同已具备管理员排期核心能力
- 甘特图主界面已支持节点/任务手工新增，不再依赖模板初始化
- 后端项目域表和接口基础较完整

### 4.2 当前偏差

- 当前菜单命名和最终业务口径还未完全统一
- `progress-config` 需要正式更名并收敛为“里程碑模板管理”
- 当前还没有独立的：
  - `任务填报`
  - `计划与日志`
  - `延期申请`
  - `项目分析看板`
- 甘特管理页中仍有部分能力边界需要继续收口，只保留管理员计划制定与排期调整相关操作
- 分析中心尚未开始建设

## 6. 页面职责定义

### 6.1 项目台账

职责：

- 按表格查看项目清单
- 项目新增、编辑、删除
- 项目经理、项目类型、计划周期、预算等主档维护
- 项目初始化入口

不承担：

- 可视化排期编辑
- 成员执行填报

### 6.2 里程碑模板管理

职责：

- 维护项目类型
- 维护节点模板
- 维护任务模板
- 维护模板排期

说明：

- 这就是现有 `project-type-management-page.tsx` 的正式业务定位
- 无需再新增第二套“里程碑模板管理”

### 6.3 排期协同

职责：

- 管理员可视化制定项目节点/任务计划
- 节点新增、子节点新增、任务新增
- 节点与任务分配
- 甘特时间拖动、缩放、调整
- 节点/任务排期调整
- 节点延期调整入口

不承担：

- 任务接收
- 完成说明填报
- 成员延期申请填报
- 月周日计划与日志填报

### 6.4 任务填报

职责：

- 成员接收任务
- 填写完成说明
- 提交任务延期申请
- 查看个人任务执行状态

页面形态：

- 表格列表 + 右侧抽屉/弹窗表单

### 6.5 计划与日志

职责：

- 月计划
- 周计划
- 日计划
- 总结
- 日志录入

页面形态：

- 表格列表 + 填报表单

### 6.6 延期申请

职责：

- 节点延期申请
- 任务延期申请
- 申请记录查看
- 审核状态或处理状态跟踪

页面形态：

- 表格管理

### 6.7 项目分析看板

职责：

- 上半部分展示型甘特图
- 下半部分图表分析

分析内容：

- 进度分析
- 任务分析
- 成员分析
- 费用分析

## 7. 分阶段实施计划

### 第一阶段：菜单与职责收口

目标：

- 先把现有页面命名和职责对齐，不再继续混用旧概念

实施项：

- 将 `项目类型管理` 正式调整为 `里程碑模板管理`
- 将 `project-gantt-workspace` 的业务名称固定为 `排期协同`
- 移除或弱化当前 `overview` 的独立工作区定位
- 在左菜单中补齐占位菜单：
  - `任务填报`
  - `计划与日志`
  - `延期申请`
  - `项目分析看板`

交付结果：

- 菜单结构和业务边界与本文档一致

### 第二阶段：完善里程碑模板管理

目标：

- 把现有节点模板页完全固化为里程碑模板管理页

实施项：

- 整理页面文案、字段命名
- 补模板头、模板节点、模板任务、模板排期的完整操作闭环
- 补模板启停、排序、说明等基础能力

交付结果：

- 完整可用的里程碑模板管理页

### 第三阶段：完善排期协同

目标：

- 让排期协同只服务于管理员计划制定与调整

实施项：

- 继续完善左树交互
- 继续完善甘特条块视觉与对齐
- 完善节点/任务新增、分配、改期、延期调整
- 清理不应留在此页的填报型入口

交付结果：

- 稳定可用的管理员甘特计划工作台

### 第四阶段：建设任务填报

目标：

- 将成员执行类操作从甘特页中独立出来

实施项：

- 新建 `任务填报` 页面
- 支持：
  - 任务接收
  - 完成说明
  - 延期申请
  - 执行状态跟踪

后端复用：

- `task`
- `report`
- 后续如有必要补专门申请单据表

### 第五阶段：建设计划与日志

目标：

- 将月周日计划和日志形成独立业务页

实施项：

- 新建 `计划与日志` 页面
- 分为：
  - 月计划
  - 周计划
  - 日计划
  - 总结
  - 日志

后端优先复用：

- `plan`
- `report`

必要时补：

- 计划类型字段
- 日志维度字段

### 第六阶段：建设延期申请

目标：

- 把延期从“甘特调整动作”扩展成“可独立流转的申请记录”

实施项：

- 新建 `延期申请` 页面
- 节点延期和任务延期统一管理
- 形成申请记录、处理状态、处理结果

后端建议：

- 第一版可先在现有 `report` 或新增申请表承接
- 第二版视需要接入审批流

### 第七阶段：建设项目分析看板

目标：

- 按你的要求建设“上甘特、下图表”的分析中心

实施项：

- 上半部分：
  - 展示型甘特图
  - 支持项目维度筛选
- 下半部分：
  - 进度分析
  - 任务分析
  - 成员分析
  - 费用分析

输出形态：

- 统计卡片
- 趋势图
- 柱状图
- 饼图
- 汇总表

## 8. 推荐执行顺序

建议按以下顺序实施：

1. 菜单与页面职责收口
2. 里程碑模板管理完善
3. 排期协同完善
4. 任务填报建设
5. 计划与日志建设
6. 延期申请建设
7. 项目分析看板建设

原因：

- 先把基础配置和管理员排期工作台稳定下来
- 再逐步拆出成员侧填报能力
- 最后做展示型分析中心，避免前面业务口径未稳就开始堆 BI

## 9. 本阶段结论

当前项目管理模块已经具备：

- 项目台账基础能力
- 里程碑模板管理基础能力
- 排期协同基础能力
- 项目域后端接口基础能力

下一阶段不应继续无边界扩展当前甘特页，而应按本文档完成“职责收口 + 菜单补齐 + 分阶段实现”。

本文档作为后续页面调整、接口补充和功能验收的执行基线。
